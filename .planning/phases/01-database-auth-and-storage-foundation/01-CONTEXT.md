# Phase 1: Database, Auth, and Storage Foundation - Context

**Gathered:** 2026-06-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Set up core database schemas (Supabase PostgreSQL), authentication (Supabase Auth), and object storage (Cloudflare R2) integration for video uploads and metadata storage.

</domain>

<decisions>
## Implementation Decisions

### Storage and Video Upload Flow
- **D-01:** Upload videos directly from the browser (Next.js) to Cloudflare R2 storage using Pre-signed URLs generated by the FastAPI backend. This bypasses backend proxying to avoid file size limitations and request timeouts for 1-hour videos.
- **D-02:** When the upload completes, the frontend sends metadata (name, storage path) to create a new database record in `public.videos` table.

### User Authentication
- **D-03:** Use standard Email and Password authentication provided out-of-the-box by Supabase Auth. No social logins (Google/GitHub OAuth) are needed for the initial v1.0.

### Database Schema and Preferences
- **D-04:** Modify the user profiles schema to store subtitle/legend configuration preferences:
  - Font color (hex/color code)
  - Font size (number/size key)
  - Font style (standard, outline, etc.)
- **D-05:** Establish base schemas for:
  - `public.profiles`: Stores user settings, full name, credits, and active preferences.
  - `public.videos`: Tracks upload logs and processing jobs status.
  - `public.clips`: Stores metadata and Cloudflare R2 links for processed vertical clips.

### the agent's Discretion
- Exact naming schema of R2 buckets and folders (recommended: `videos/` and `clips/` partitioned by user UUID).
- Session length and security rules for token storage.
- Storage eviction policy for temp files in backend.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `backend/src/supabase_client.py`: Can be updated to support the new tables schema and handle storage connections.
- `supabase/schema.sql`: Provides SQL declarations for PostgreSQL table definitions and Row Level Security (RLS) triggers.

### Established Patterns
- Supabase Row-Level Security (RLS) is applied to all public tables using `auth.uid() = user_id`.
- FastAPI backend exposes RESTful endpoints; we will add a route for generating R2 pre-signed URLs.

### Integration Points
- Frontend client joins with Supabase JS client for Auth state monitoring.
- Next.js forms connect to Supabase Auth endpoints for registration and login.
- R2 API integration in FastAPI backend using `boto3`.

</code_context>
